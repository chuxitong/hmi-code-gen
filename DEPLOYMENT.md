# Развёртывание UI2Code^N на локальной машине

Здесь описано, как я поднимаю настоящую визуально-языковую модель UI2Code^N у себя в проекте, какие подводные камни встретились и что делать, если у вас другая конфигурация железа.

## Что нужно по железу

Модель UI2Code^N — это 9 миллиардов параметров поверх GLM-4.1V-9B-Base. В bf16 веса занимают около 18 ГБ. Для комфортной работы нужна либо серверная карта на 24 ГБ (RTX 3090/4090, A5000+), либо две карты на 12 ГБ в параллели. На ноутбуке с 6 ГБ VRAM приходится идти по компромиссам, они описаны ниже.

У меня в тестах: RTX 3060 Laptop (6 ГБ), 64 ГБ DDR5, 140+ ГБ свободного диска. CUDA 12.3 драйвер, CUDA 12.4 runtime через torch.

## Как я ставлю окружение

Поднимаю отдельное виртуальное окружение в `local-service/.venv`. В нём идут ключевые пакеты: PyTorch 2.4.1+cu124 (через индекс `https://download.pytorch.org/whl/cu124`), `transformers==4.57.1` (именно эта версия работает с UI2Code^N на момент проверки), `accelerate`, `bitsandbytes`, `huggingface_hub`, а также `fastapi`, `uvicorn`, `pillow`, `playwright`. После установки `playwright` нужно отдельно выполнить `playwright install chromium`, иначе headless-браузер не запустится.

## Как я качаю веса

Модель лежит на Hugging Face по адресу `zai-org/UI2Code_N`. Четыре шарда safetensors плюс токенизатор и конфиг — в сумме около 18 ГБ. Качаю через `huggingface_hub.snapshot_download`. Из России и Китая прямой HF часто медленный, поэтому я использую зеркало `hf-mirror.com` через переменную окружения.

Команда скачивания, которую я у себя запускаю:

```bash
set HF_ENDPOINT=https://hf-mirror.com
python -c "from huggingface_hub import snapshot_download; snapshot_download('zai-org/UI2Code_N', local_dir='d:/hf_models/UI2Code_N', max_workers=4)"
```

У меня полная выгрузка на домашнем канале заняла около 15 минут. На лабораторной сети или через VPN это могло бы быть быстрее.

## Как я загружаю модель

В проекте лежит тонкая обёртка `local-service/model_wrapper.py`, которая инкапсулирует всё про загрузку. Она принимает переменную окружения `UI2CODEN_MODEL_ID` (по умолчанию `zai-org/UI2Code_N`), умеет собирать сообщения в формате `apply_chat_template`, декодирует вывод и чистит маркеры вроде ``` ```html ``` ```. Интерфейс совпадает с детерминированным заместителем (`rule_based_model.py`), поэтому сервис и скрипты ничего про саму модель не знают.

Сейчас я грузю модель одним из двух способов. Первый — 4-битная квантизация через `bitsandbytes` (NF4, double-quant, compute dtype bf16). Если модель целиком не помещается в GPU, `accelerate` автоматически раскидает часть слоёв по оперативной памяти — этот режим включается флагом `llm_int8_enable_fp32_cpu_offload=True` в `BitsAndBytesConfig`. На моём 6 ГБ GPU эта конфигурация работает неустойчиво из-за известного бага bitsandbytes + accelerate вокруг meta-тензоров при сохранении state_dict квантизованных слоёв. Поэтому как практически рабочий вариант на 6 ГБ я использую второй способ — чистый CPU-инференс в bf16. Это устойчиво, но медленно: на моём ноутбуке первый запрос занимает несколько минут из-за пагинации весов через mmap.

Если у вас нормальный GPU на 24 ГБ, всё работает прямо из коробки в bf16 с `device_map="auto"`, без квантизации и без CPU-offload. Тогда генерация идёт на десятках токенов в секунду.

## Как подключить модель к сервису

В `local-service/app.py` функция `get_model` при первом запросе смотрит на переменную окружения `USE_REAL_MODEL`. Если она установлена в `1`, сервис пытается импортировать `UI2CodeModel` из `model_wrapper.py` и загрузить настоящую модель. Если что-то пошло не так (нет весов, не хватает памяти, несовместимые версии), сервис молча откатывается на детерминированный заместитель и пишет об этом в лог. Это сделано специально: пайплайн не должен зависать из-за проблем с моделью, его всегда можно проверить end-to-end на заместителе.

Чтобы включить настоящую модель:

```bash
set USE_REAL_MODEL=1
cd local-service
.venv\Scripts\python.exe -m uvicorn app:app --port 8000
```

Чтобы вернуться к заместителю, достаточно снять переменную и перезапустить сервис.

## Что я реально запускал и что получил

На моём ноутбуке (RTX 3060 Laptop 6 ГБ + 64 ГБ RAM) с текущим `model_wrapper.py` модель успешно качается и загружается (`Loading checkpoint shards: 100%`), первый запрос инициируется корректно, генерация идёт, но на CPU bf16 темп ~0.2–0.4 токена в секунду, поэтому полная страница из нескольких тысяч токенов требует нескольких минут. Для воспроизводимых экспериментальных прогонов по всем восьми мокапам я использую `rule_based_model.py` — он отрабатывает за секунды и позволяет проверять весь пайплайн (плагин, сервис, рендер, пост-обработка, метрики) без зависимости от GPU. На выделенной машине с 24 ГБ VRAM те же самые скрипты отработают с настоящей моделью за минуты и выдадут итоговую таблицу таймингов, уже отражающую реальную производительность UI2Code^N.

## Короткая чек-листовая последовательность

Установить Python 3.12, создать venv в `local-service/.venv`, поставить торч под CUDA (`pip install torch==2.4.1 --index-url https://download.pytorch.org/whl/cu124`), поставить остальные зависимости (`pip install -r local-service/requirements.txt`, а также `transformers==4.57.1`, `accelerate`, `bitsandbytes`, `huggingface_hub`), установить браузер для Playwright (`playwright install chromium`), скачать веса через `snapshot_download` с зеркала, собрать плагин Figma через `npx tsc` в `figma-plugin/`. После этого при желании выставить `USE_REAL_MODEL=1` и запускать сервис и экспериментальные скрипты.
