# Глава 2. Формализация задачи, подход к решению, функциональность модуля и данные

## Формализация задачи

Чтобы не путаться в словах, удобно записать задачу в компактных обозначениях. Входом считаем растровое изображение фрейма M, полученное из Figma экспортом выделенного узла, и опциональный контекст C, составленный из двух частей: множества переменных дизайна V (цвета, отступы, радиусы, токены типографики) и множества CSS-подсказок S, взятых по иерархии выделения через `getCSSAsync`. Инструкцию пользователя на естественном языке обозначим u. Результатом работы считаем HTML/CSS-документ K, а его растеризацию во фрейме фиксированного размера (viewport 1280×720) — изображение R(K).

Три сценария, которые модуль обязан поддерживать, описываются тремя функциями. Generate — это функция G, которая по макету и контексту выдаёт начальную версию кода, K₀ = G(M, C). Refine — это функция Rᵣ, которая на каждой итерации i берёт исходный макет, текущий рендер и текущий код и выдаёт улучшенный код, Kᵢ₊₁ = Rᵣ(M, R(Kᵢ), Kᵢ, C). Edit — функция E, применяющая текстовую инструкцию, Kᵢ₊₁ = E(Kᵢ, u, C). Все три функции инкапсулированы в одной визуально-языковой модели, но разделены по ролям — этого достаточно, чтобы пайплайн оставался понятным.

Критерий приемлемости результата намеренно оставлен человекоориентированным. Оператор смотрит на пару «референс — рендер», оценивает совпадение по цветовой палитре, типографике, иерархии, отступам и наличию ключевых элементов, и либо принимает результат, либо запускает ещё одну итерацию Refine или точечную правку Edit. Жёсткий пиксельный критерий здесь нерелевантен, потому что модель не обязана повторять пиксели, а должна сохранить смысл интерфейса.

Ограничения задачи, которые мы намеренно наложили: макет подаётся с viewport 1280×720, итог должен быть единым HTML-документом с inline-стилями и без внешних ресурсов, все тяжёлые вычисления выполняются снаружи плагина (внутри плагина никакой модельной логики нет).

## Общий подход к решению

Решение устроено как распределённая система из трёх компонентов, соединённых предсказуемыми контрактами. Первый компонент — плагин в Figma, отвечающий за пользовательский сценарий и извлечение данных из макета. Он обращается к сервису только по HTTP, ничего «умного» внутри себя не выполняет. Второй — локальный сервис на FastAPI, принимающий запросы от плагина и вызывающий модель. Третий — сама модель, реализующая три режима, и рядом с ней вспомогательный рендер HTML/CSS в PNG через headless-браузер.

Такая развязка даёт сразу несколько выгод. Модель можно заменить на новую версию без правок в плагине. Плагин можно тестировать вообще без модели, подключив к сервису детерминированный заместитель с тем же интерфейсом. Рендер живёт отдельно, и его легко использовать как для превью внутри плагина, так и для воспроизводимых экспериментов в командной строке.

## Функциональность программного модуля

В тексте ниже намеренно нет кода — только инженерное описание поведения.

### Generate Code

Пользователь в Figma выделяет один фрейм и жмёт кнопку Generate Code. Плагин проверяет корректность выделения (ровно один фрейм подходящего типа), экспортирует его в PNG с двукратным масштабом, при включённых галках собирает переменные и CSS-подсказки по выделенной иерархии. Сформированный запрос уходит на сервис на эндпоинт `/generate`. Сервис вызывает модель в режиме UI-to-code, применяет прикладные правила (одна страница, inline-стили, семантические теги), и возвращает плагину пару «код + превью PNG». Плагин показывает код в текстовом поле, превью в области предпросмотра и разрешает копирование кода.

### Make It Closer to the Mockup

Плагин хранит исходный PNG макета с момента генерации. При нажатии кнопки Make It Closer to the Mockup он запрашивает у сервиса текущий рендер (`/render` по текущему коду), после чего отправляет на `/refine` три вещи: референсный PNG, рендер-PNG и сам код. Сервис вызывает модель в режиме UI-polishing, получает обновлённый код, возвращает его плагину. Плагин заменяет текст в textarea и обновляет превью. Итерацию можно повторять, но в UI зафиксировано ограничение в три шага — дальше эффект становится незначимым или отрицательным.

### Edit by Request

Пользователь вводит короткую инструкцию на естественном языке и жмёт Apply Edit. Плагин шлёт на `/edit` текущий код и инструкцию, сервис вызывает `model.edit`, и возвращает исправленную версию кода. Типовые инструкции, которые прототип отрабатывает устойчиво: «сделай кнопку вторичной», «увеличь заголовок», «выдели блок аварий», «уменьши отступ над трендом», «сделай рамку карточек толще».

### Render (служебный)

Эндпоинт `/render` принимает HTML и viewport, возвращает PNG. Он используется и как вспомогательная функция внутри других эндпоинтов (`/generate` и `/refine` включают превью в ответ), и как инструмент для экспериментальных скриптов (`baseline-tests/run_full_evaluation.py` и его соседи).

### Опциональный контекст

В UI плагина есть две галки. Галка «Include design variables» включает в запрос локальные переменные Figma, собранные через `figma.variables.getLocalVariablesAsync`. Галка «Include CSS hints» добавляет в запрос результаты `getCSSAsync` по выделению и его ближайшим детям (ограничено десятью элементами, чтобы не раздувать запрос). Эти каналы работают независимо и в Generate, и в Refine, и в Edit.

## Архитектура и контракты

Сервис предоставляет четыре HTTP-эндпоинта. `/generate` принимает image_base64, опциональные variables и css_hints, возвращает поле code, опциональный preview_base64 и опциональное текстовое пояснение. `/refine` принимает reference_image_base64, rendered_image_base64, current_code и опциональный контекст, возвращает те же три поля. `/edit` принимает current_code, instruction и опциональный контекст, возвращает обновлённый код. `/render` принимает html_code и viewport, возвращает image_base64. Все ответы нормализованы pydantic-моделями, поэтому клиентская сторона работает с предсказуемыми структурами без проверок на null.

Разделение ответственности: плагин отвечает за UX и данные из Figma, сервис — за нормализацию запроса и вызов модели, модель — за саму семантическую задачу, рендер — за воспроизведение кода как изображения. Заменяемость любого слоя сохраняется. В частности, переключение между настоящей UI2Code^N и детерминированным заместителем меняет одну переменную окружения `USE_REAL_MODEL` и не требует правок ни в плагине, ни в остальных скриптах.

## Процесс итеративного взаимодействия

Типовая сессия выглядит так. Оператор выбирает фрейм, жмёт Generate Code, получает начальную версию. Один или два раза нажимает Make It Closer to the Mockup, подтягивая рендер к референсу. Затем при необходимости делает одну-две локальные правки через Edit by Request. После этого копирует итоговый HTML/CSS в свой инженерный workflow. Весь цикл укладывается в несколько минут на простых экранах и до получаса на сложных.

## Данные

Для обучения модели данные не нужны: UI2Code^N используется в режиме инференса без дообучения. Данные требуются для тестирования и демонстрации.

### Внутренний тестовый набор

В рамках работы собрано восемь мокапов промышленных HMI с разной сложностью. Два простых — обзор состояния оборудования и таблица аварий с подтверждениями. Четыре средней сложности — мониторинг трендов, операторская панель управления, обзор производственной линии, энергодашборд. Один «средне-сложный» — синоптика резервуарного парка с уровнями, клапанами и трубопроводами. Один действительно сложный — экран управления рецептом партии с последовательностью шагов, таблицей параметров и временной осью.

Для каждого макета зафиксирован тип экрана, типовая индустриальная задача, уровень сложности и короткое обоснование включения. Подробности в `mockups/mockup-index.md`. Все макеты сохранены в Figma и экспортированы в PNG для воспроизводимых прогонов. В качестве источников для базовых layout-ов использованы Themesberg Admin Dashboard Template и Sneat Free UI Kit. Визуальные промышленные ориентиры взяты из Siemens HMI Template Suite, публичного демо Ignition SCADA, FUXA и JointJS SCADA/HMI Demo.

### Внешние датасеты

Большие публичные корпуса — WebSight и Web2Code — в настоящей работе напрямую не используются для обучения, потому что речь о готовой модели в режиме инференса. Они присутствуют в обзоре литературы как методологический контекст. Design2Code относится к той же категории: как бенчмарк он задаёт подход к оценке, но для промышленного HMI у меня собственный набор, а не подмножество Design2Code.

## Методика оценки

Оценка строится вокруг воспроизводимых метрик времени и качественной классификации ошибок. По каждому мокапу и каждой фазе (генерация, две итерации refine, правка) делается по три триала для усреднения. Замеряется время до первого сгенерированного результата и время до результата, принятого как приемлемый. Подсчитывается число итераций Refine до приемлемости. Для типичных ошибок ведётся классификация по семи категориям, взятым из реальных прогонов: отступы, типографика, выравнивание, визуальная иерархия, стиль кнопок, пропуск мелких элементов, искажение графиков.

Отдельный раздел посвящён влиянию контекста. Для выбранного экрана последовательно прогоняются три конфигурации (только картинка, картинка + переменные, картинка + переменные + CSS-подсказки), сравниваются визуально и по коду. Итог — рекомендация по дефолтной конфигурации контекста для промышленного HMI.

## Итог

В главе зафиксирована формальная запись задачи, общая архитектура, подробно описана функциональность трёх режимов без включения исходного кода, описаны данные и методика оценки. Всё это образует инженерную и методологическую основу для экспериментальной части, изложенной в главе 3.
